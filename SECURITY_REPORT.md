# SECURITY REPORT

Дата: 2026-06-16

## Итог

Критичных уязвимостей в реализованной логике продажи не обнаружено. Основные риски double sale, double spend, callback-подмена и повторный `successful_payment` закрыты на уровне payload-подписи, транзакций, блокировок и уникальных ограничений.

## Проверенные области

### Telegram Security

- Админ-доступ проверяется через `ADMIN_IDS` и активные записи `admins`.
- Пользовательские callback не дают прямого доступа к чужим заказам.
- Покупки пользователя выбираются только по `User.telegram_id`.
- `payment_payload` подписан HMAC и привязан к `order_id`, `user_id`, `product_id`, `nonce`.
- Подмена суммы или валюты переводит заказ в ошибку и не выдает код.
- Повторный `successful_payment` возвращает уже выданный код без новой выдачи.

### Database Security

- SQL Injection: используются SQLAlchemy expressions, raw SQL в сервисах и handlers не применяется.
- Race Condition: выдача кода выполняется внутри транзакции.
- Double Sale: выбор цифрового товара использует `FOR UPDATE SKIP LOCKED`.
- Double Spend: charge id платежа уникален.
- Lost Update: заказ блокируется `FOR UPDATE` перед сменой статуса.
- Dirty Read: PostgreSQL default isolation `READ COMMITTED` совместим с row-level locking; для этой схемы достаточно.

### API Security

- `callback_data` ограничен по длине.
- Текстовые поля проходят проверку длины.
- Пустые цифровые товары отбрасываются.
- Слишком длинные цифровые товары отбрасываются.
- Валюта нормализуется к ISO-4217 виду.

### Admin Security

В `audit_logs` пишутся:

- создание категорий
- создание подкатегорий
- создание товаров
- включение и отключение сущностей
- загрузка цифровых товаров
- изменение системных текстов
- блокировка и разблокировка пользователей

## Найденные проблемы и исправления

1. Риск двойной выдачи при повторном `successful_payment`.
   Исправлено: статус `paid` обрабатывается идемпотентно.

2. Риск продажи одного кода двум заказам.
   Исправлено: `FOR UPDATE SKIP LOCKED` и уникальная связь `digital_items.order_id`.

3. Риск reuse одного Telegram charge id.
   Исправлено: уникальные ограничения в `payments`.

4. Риск подмены payload.
   Исправлено: HMAC-подпись и сверка со stored payload.

5. Риск доступа к чужим покупкам.
   Исправлено: выборки привязаны к `telegram_id`.

6. Риск покупки заблокированным пользователем.
   Исправлено: проверка перед созданием заказа и pre-checkout.

7. Риск покупки отключенного товара.
   Исправлено: проверка товара, категории и подкатегории.

8. Риск выдачи при пустом складе.
   Исправлено: проверка остатков до счета и повторная атомарная проверка после оплаты.

9. Риск массового создания заказов.
   Исправлено: Redis rate limit для заказов.

10. Риск callback flood.
    Исправлено: Redis middleware.

11. Риск message flood.
    Исправлено: Redis middleware.

12. Риск SQL injection при поиске кодов.
    Исправлено: SQLAlchemy `ilike` с bound parameter.

13. Риск слишком длинных callback.
    Исправлено: ограничение длины.

14. Риск загрузки дублей из файла.
    Исправлено: deduplicate preserve order.

15. Риск загрузки дублей из БД.
    Исправлено: проверка existing values и unique constraint.

16. Риск потери истории админ-действий.
    Исправлено: `audit_logs`.

17. Риск запуска webhook без URL.
    Исправлено: явная ошибка конфигурации.

18. Риск неготовности БД.
    Исправлено: healthcheck PostgreSQL и Redis в compose.

19. Риск неявного host network.
    Исправлено: отдельная bridge network.

20. Риск повреждения VPN при деплое.
    Исправлено: деплой-стратегия не меняет firewall, VPN, routing и host network.

## Остаточные риски

- Telegram Payments final settlement зависит от выбранного провайдера.
- Для webhook-режима нужен отдельный ASGI runner и TLS/reverse proxy.
- Админ-команды в чате удобны для MVP, но для большой команды лучше добавить полноценные FSM-формы и роли.
- Значения цифровых товаров хранятся в БД открытым текстом, потому что должны выдаваться пользователю. Для особо чувствительных кодов рекомендуется шифрование на уровне приложения.
- Рассылки могут упереться в лимиты Telegram, для больших баз нужен очередной worker.

## Рекомендации по резервному копированию

- Ежедневный `pg_dump` с хранением минимум 14 дней.
- Еженедельный полный backup volume PostgreSQL.
- Отдельно хранить `.env` в секретном хранилище.
- Перед обновлениями сохранять `docker-compose.yml`, `.env`, список контейнеров и список systemd-сервисов.
- Периодически тестировать восстановление на staging-сервере.

## Рекомендации по мониторингу

- Алерты на падение контейнера `bot`.
- Алерты на рост заказов `error`.
- Алерты на остатки кодов ниже порога.
- Алерты на ошибки рассылки.
- Мониторинг диска PostgreSQL volume.
- Логирование в JSON уже включено через `structlog`.
